## Addendum: source-of-truth investigation, 2026-05-22

The original draft of this doc treated `signalapp/libsignal` as a black box and
assumed direct-on-libsignal meant building the WebSocket transport, the
provisioning handshake, the HTTPS endpoints, and the chat-protocol framing
ourselves. The Architect's review flagged that as the load-bearing risk.

A subsequent read of the actual `signalapp/libsignal` source (locally at
`~/repos/signalapp/libsignal`, version `0.94.1`) and of `AsamK/signal-cli`'s
build configuration (locally at `~/repos/AsamK/signal-cli`) substantially
changes the picture. The findings:

1. **`libsignal-net` (`rust/net/`) provides the chat-service WebSocket client**,
   including the request/response framing on top of WSS via
   `tokio-tungstenite`. The chat protobuf types (`MessageProto`,
   `RequestProto`, `ResponseProto`) and the environment configuration
   (prod/staging service URLs, route selection) are first-class members of
   the crate. We don't write the WebSocket layer; we use it.

2. **`libsignal-net` also provides the provisioning WebSocket client**, via
   `chat::server_requests::ProvisioningEvent`. The consumer hooks into two
   events: `ReceivedAddress { address }` (the opaque address string we
   bake into the QR code) and `ReceivedEnvelope { envelope: Vec<u8> }` (the
   encrypted ProvisionMessage bytes from the primary). The WebSocket
   framing, the `/v1/address` and `/v1/message` request routing, and the
   ack handling are all in libsignal.

3. **`libsignal-net-chat` (`rust/net/chat/`) provides typed API endpoints**
   for everything the chat WebSocket can do. Specifically:
   `UnauthenticatedChatApi::get_pre_keys(target, device) -> (IdentityKey,
   Vec<PreKeyBundle>)` (prekey-bundle fetch),
   `AuthenticatedChatApi::send_message(destination, contents, ...)` (1:1
   send), and **`AuthenticatedChatApi::send_sync_message(timestamp,
   contents, ...)` - exactly the Note-to-Self path borg needs**. All
   async-native, takes Signal's modern types (`ServiceId`, `DeviceId`,
   `PreKeyBundle`, `IdentityKey`) from `libsignal-protocol`.

4. **signal-cli does NOT use `libsignal-net` or `libsignal-net-chat`.** Its
   `lib/build.gradle.kts` depends on `com.github.turasa:signal-network` (a
   JVM port of Signal's legacy `signal-service-android` library) plus
   libsignal-client for protocol crypto. The version file `libsignal-version`
   (`0.92.1`) tracks only the libsignal-client artifact. The reason is
   timing: `libsignal-net-chat` is a 2025 addition; signal-cli was built
   when only the JVM-port path existed and has not been migrated. **We
   are not constrained to copy signal-cli's stack.** signal-rs can target
   the newer, native Rust layer signal-cli does not.

5. **The one remaining gap: `ProvisioningCipher`.** A repo-wide search of
   `signalapp/libsignal` (Rust, Java, Swift, Node) found NO
   `ProvisioningCipher` anywhere. `libsignal-net`'s `ProvisioningEvent`
   hands the consumer opaque encrypted bytes; the X3DH-style key derivation
   plus AES-CBC + HMAC-SHA256 that the primary's Signal client used to
   encrypt the envelope is the consumer's responsibility. In signal-cli's
   stack that crypto lives in `turasa/signal-network`. For signal-rs we
   port it from the well-specified `signal-service-android`
   `ProvisioningCipher.java` reference - a few hundred lines of Rust,
   standalone, crypto-only, no networking, no protocol design. This is
   the one piece we genuinely write ourselves.

6. **Version pin needs to lead signal-cli's.** signal-cli pins libsignal
   `0.92.1`, but `libsignal-net-chat` may not be present (or fully fledged)
   at that tag. We pin to `0.94.x` or later - Phase 2 verifies the minimum
   commit that includes the `libsignal-net-chat` API surface we depend on.

**Net effect on the design:** Branch A (direct-on-libsignal is feasible)
holds, with one clearly-bounded crypto port as the only piece we own.
The Architect's worst-case scenario - reverse-engineering Signal's
WebSocket protocol from scratch - is ruled out empirically. Phases 5
(was: chat WebSocket from scratch) and 6 (was: send pipeline plumbing)
collapse to thin wrappers over `libsignal-net::chat::*` and
`libsignal-net-chat::api::*` respectively. A new Phase 4 (ProvisioningCipher
port) joins the plan to fill the one real gap. The Risks table's "libsignal
has no networking layer" entry downgrades from High/High to Low/Low. The
v0.1 design holds as written below, but with the phase-shape changes the
phase headings now reflect.

## Summary

`signal-rs` is a native-Rust replacement for `AsamK/signal-cli`, providing both a
command-line tool and a consumable Rust library that sit directly on
`signalapp/libsignal`. **The v0.1 slice is the minimum surface required to
unblock `scottidler/second-brain`'s borg Note-to-Self ingest path**: link as a
secondary device, receive Signal envelopes, send a 1:1 text message. Every other
signal-cli subcommand is deferred to later versions; the destination is full
parity, the starting point is a working borg integration via in-process Rust
crate (Shape B per `~/repos/scottidler/second-brain/docs/signal-as-borg-transport.md`).

## Problem Statement

### Background

borg's existing inbound transport is Telegram, integrated via `teloxide` as an
in-process Rust crate. The cited reference doc
(`scottidler/second-brain/docs/signal-as-borg-transport.md`) lays out the option
space for adding Signal as a peer inbound transport. The option space resolves
to two integration shapes:

- **Shape A** - Out-of-process `signal-cli` daemon (JVM, ~250 MB RSS, JSON-RPC
  over Unix socket). The path described in
  `tatari-tv/thoughts/directors/scott.idler/2026-05-21-signal-ingest-via-linked-device.md`.
- **Shape B** - In-process Rust crate. Matches `teloxide`'s posture for
  Telegram. Either `presage` today, or a future Rust-native client whose surface
  was nicknamed `signal-rs` in the Slack thread with Keegan.

This document is the design for that nicknamed project.

### Problem

borg cannot integrate Signal cleanly while the only mature client is a Java
daemon:

- Shape A pays a JVM tax (~250 MB RSS, multi-second cold start) and requires
  two systemd units to manage instead of one.
- Shape A requires borg to talk JSON-RPC over a Unix socket - extra serialization
  hop, extra failure mode, structurally divergent from the in-process posture
  Telegram already uses.
- Java means no Rust consumer can directly call into the Signal client. Every
  consumer pays the socket cost, even when one process could do the work.

`presage` (Rust crate on top of `libsignal`) solves Shape B today for borg's
specific subset (1:1 conversations + sync messages), but is third-party
maintained, has had API breaks between minor versions, and does not aim to
replace signal-cli's full surface.

`signal-rs` exists to be the project the ecosystem is missing: a native-Rust
Signal client that is **both** a CLI replacement for signal-cli AND a
library Rust consumers can `cargo add`. v0.1 ships only the slice borg needs;
later versions reach signal-cli parity by accretion.

### Goals

- **Unblock borg's Note-to-Self ingest path via `cargo add signal-rs`** -
  second-brain depends on signal-rs as a library, in-process, no daemon.
- Expose three load-bearing library functions: `link`, `receive`, `send`,
  matching the operations enumerated in the reference doc § "Is presage adequate".
- Surface `SyncMessage::Sent` envelopes with the `destination` field intact, so
  borg can apply the load-bearing `destination == own_number` filter that
  distinguishes Note-to-Self from "every outbound message to anyone."
- Provide a CLI entry point (`signal-rs link`) for the one operation a daemon
  can't do - the interactive QR-scan ritual that adds the host as a secondary
  device on the user's existing account.
- Be structured so signal-cli's wider surface (groups, attachments, daemon
  mode, registration, profile, identity, sticker packs) can be added in later
  versions without breaking the v0.1 surface.

### Non-Goals

For v0.1, explicitly OUT of scope:

- **JSON-RPC daemon mode.** Shape B exclusively. The signal-cli wire-compatible
  daemon arrives in a later version once the library is solid.
- **Attachments.** borg's Note-to-Self ingest path is URLs and text. Attachments
  are deferred.
- **Groups.** Single-operator, single Note-to-Self conversation. The
  `dataMessage` envelope path stays available structurally (the envelope type
  exposes both variants) but no group-specific operations.
- **Registration of new accounts.** signal-rs is always a secondary device
  against an existing primary; the `register` / `verify` / `setPin` family is
  unimplemented.
- **Profile, contact sync, sticker packs, typing indicators, read receipts,
  remote-delete, sealed-sender opt-in, identity-key trust management,
  multi-account.** All deferred.
- **Headless link flow.** The "device-registration pattern signal-cli doesn't
  support" Keegan flagged is a separate later feature. v0.1 requires a QR scan
  from the primary, identical to signal-cli.
- **Drop-in CLI parity with signal-cli's subcommand vocabulary.** The CLI
  exposes `link` plus a `send` and `receive --once` helper for development
  smoke-testing. Other signal-cli subcommands are added later.

## Proposed Solution

### Overview

Single Cargo crate, two artifacts: a `[lib]` (`signal_rs`) for in-process
consumption by Rust callers, and a `[[bin]]` (`signal-rs`) as a thin clap
shell over the library. Built directly on `signalapp/libsignal` as a git
dependency, composing three of its workspace crates:

- `libsignal-protocol` for crypto (Double Ratchet, X3DH, PQXDH, sender
  keys, sealed sender), storage traits, and types (`ServiceId`,
  `DeviceId`, `PreKeyBundle`, `IdentityKey`).
- `libsignal-net` for the chat-service WebSocket client and the
  provisioning WebSocket client (`ProvisioningEvent` event stream).
- `libsignal-net-chat` for typed API endpoints:
  `AuthenticatedChatApi::send_sync_message` (the Note-to-Self path),
  `AuthenticatedChatApi::send_message` (1:1), and
  `UnauthenticatedChatApi::get_pre_keys` (prekey-bundle fetch).

The one piece signal-rs implements itself is `ProvisioningCipher` (the
X3DH-style key derivation plus AES-CBC + HMAC-SHA256 that decrypts the
encrypted ProvisionMessage payload arriving over the provisioning
WebSocket). libsignal hands us opaque bytes; we own the decryption. The
reference port is `signal-service-android`'s `ProvisioningCipher.java`.

Storage backed by `sqlx` with the SQLite backend (file-based, single-file,
no separate server) for the device's identity, registration IDs, Double
Ratchet session state, and prekey state. Async runtime: `tokio` (per
`rules/rust.md`).

Storage choice rationale: SQLite is the more battle-tested embedded store
than sled, and `presage`'s recent direction has moved off sled toward sqlx
for the same reason. We use `sqlx` (not `rusqlite`) because sqlx is
async-native; the Client multiplexes an async receive loop and an async
send pipeline against the same storage, and sqlx's `SqlitePool` handles
the concurrent-access shape cleanly without an external worker-thread
wrapper. The `Store` trait is abstract enough that swapping backends
later is a contained change; we lead with sqlx+SQLite to minimize churn
risk and match presage's direction.

**Concurrency model.** A single `sqlx::SqlitePool` is owned by `Client`
for its lifetime. The receive loop and send pipeline both check out
connections from the pool. SQLite serializes writes at the file level
regardless of pool size, which gives us the natural transaction ordering
libsignal's storage-trait calls require - within a single decrypt or
send, the storage trait calls must observe each other's writes
atomically, which we get for free by wrapping each logical operation in
a `sqlx::Transaction`. Compile-time query checking via `sqlx::query!`
is opt-in; for v0.1 we use runtime `sqlx::query(...)` to avoid the
build-time-needs-live-DB friction and reach for `query!` only if a
specific query benefits from compile-time validation.

Crate layout uses the **Rust 2018+ module style throughout** (per
`rules/rust.md`): the module entry point is `foo.rs`, with submodules living
in `foo/` alongside it. No `mod.rs` files anywhere.

Key invariants the scaffold already enforces and v0.1 preserves:

- **CLI types stay in the binary.** `mod cli;` is declared from
  `main.rs`, not `lib.rs`. Library consumers (borg, future loki-token
  work, etc.) never depend on clap. The doc had drafted this
  incorrectly; the refreshed scaffold ships the right shape.
- **`config` is public from the lib.** The lib re-exports `Config`
  directly. Both `main.rs` and library consumers reach for the same
  type.
- **There is no `transport.rs`** in signal-rs's own source tree. The
  chat-service WebSocket, the provisioning WebSocket, and the typed
  API client (send, send_sync_message, get_pre_keys) are all consumed
  from `libsignal-net` and `libsignal-net-chat`. `client.rs` holds a
  `libsignal_net::chat::ChatConnection` (or equivalent typed handle)
  and orchestrates the receive loop, the storage transactions, and
  the broadcast channel that feeds `Client::receive`.

### Data Model

**Public types** (re-exported from `signal_rs::lib`):

```rust
#[non_exhaustive]
pub enum Envelope {
    /// Someone else sent the user a message.
    DataMessage {
        source: String,           // sender E.164
        timestamp: u64,           // ms since epoch
        message: DataMessage,
    },
    /// The user sent a message from another linked device.
    /// Filter on `destination == own_number` to detect Note-to-Self.
    SyncMessage(SyncMessage),
}

#[non_exhaustive]
pub struct DataMessage {
    pub body: Option<String>,
    pub timestamp: u64,
    // Attachments, mentions, quotes, reactions: deferred. The non_exhaustive
    // attribute keeps adding them later a non-breaking change.
}

#[non_exhaustive]
pub enum SyncMessage {
    Sent {
        destination: String,      // E.164 - load-bearing for borg's filter
        timestamp: u64,
        message: DataMessage,
    },
    // Read, Contacts, Configuration, etc.: deferred.
}

pub struct Client {
    // owns one WebSocket to Signal's chat service for the lifetime of the
    // Client. receive() subscribes to envelopes via a broadcast channel,
    // send() sends through the same WebSocket. There is only ever one
    // chat-service WebSocket per linked device (the Signal protocol does not
    // tolerate two).
    inner: Arc<ClientInner>,
}

impl Client {
    /// Open an existing state directory and connect to Signal.
    /// Fails if:
    /// - The state dir doesn't exist or the identity table is empty
    ///   (`OpenError::NotLinked` - caller should run `link` first).
    /// - The state dir exists with `link_status == IdentityPersisted` but
    ///   prekey upload never completed (`OpenError::PartiallyLinked` -
    ///   caller should run `link` again from the same state dir to
    ///   resume the prekey upload step).
    /// - Another `signal-rs` process holds the flock on `store.db`
    ///   (`OpenError::AlreadyOpen`).
    /// - The libsignal device has been deauthorized from the primary
    ///   (`OpenError::Deauthorized` - terminal; only recovery is `link`
    ///   from scratch in a fresh state dir).
    pub async fn open(state_dir: &Path) -> Result<Self, OpenError>;

    /// Subscribe to incoming envelopes. Multiple subscribers are allowed
    /// (they share the underlying WebSocket via broadcast). Slow subscribers
    /// that fall behind get a `Lagged` error; the stream resumes from the
    /// next available envelope rather than dropping silently.
    pub fn receive(&self) -> EnvelopeStream;

    /// Send a 1:1 text message. `target` is an E.164 number. To produce a
    /// Note-to-Self entry, pass the account's own number - Signal fans the
    /// outgoing message to all linked devices, surfacing it in the phone's
    /// Note to Self conversation.
    pub async fn send(&self, target: &str, body: &str) -> Result<(), SendError>;
}

/// First-time linking. Creates the state directory (mode 0700) if missing,
/// runs the Signal provisioning protocol, persists identity, uploads initial
/// prekeys. Blocks until the primary device scans the displayed QR or until
/// `PROVISIONING_TIMEOUT` elapses.
///
/// `display_qr` is a callback the caller invokes to actually render the
/// provisioning URI - the CLI renders it as a terminal QR; a library
/// consumer might forward it elsewhere. This keeps `link` free of any
/// terminal-rendering dependency.
pub async fn link(
    state_dir: &Path,
    device_name: &str,
    display_qr: impl FnOnce(&str),
) -> Result<LinkOutcome, LinkError>;

pub struct LinkOutcome {
    pub account_number: String,
    pub device_id: u32,
}
```

**Storage schema** (SQLite tables managed via `sqlx::SqlitePool`, file at `<state_dir>/store.db`):

| Table | Columns | Purpose |
|---|---|---|
| `identity` | `key TEXT PRIMARY KEY, value BLOB NOT NULL` | Singleton rows for `identity_keypair`, `registration_id`, `account_number`, `device_id`, `link_status` (values: `IdentityPersisted`, `Linked`) |
| `sessions` | `address TEXT PRIMARY KEY, record BLOB NOT NULL` | Double Ratchet state per peer (`address` is `<E.164>.<device-id>`) |
| `prekeys` | `id INTEGER PRIMARY KEY, record BLOB NOT NULL` | Unsigned one-time prekeys |
| `signed_prekeys` | `id INTEGER PRIMARY KEY, record BLOB NOT NULL` | Signed prekeys |
| `kyber_prekeys` | `id INTEGER PRIMARY KEY, record BLOB NOT NULL` | Post-quantum prekeys (PQXDH) - confirm libsignal v0.92.x requires these |

State directory layout:

```
$XDG_DATA_HOME/signal-rs/        (~/.local/share/signal-rs on Linux,
                                  ~/Library/Application Support/signal-rs on macOS)
├── store.db                     # SQLite single-file database (sqlx-managed, mode 0600)
├── store.db-wal                 # SQLite WAL journal
├── store.db-shm                 # SQLite WAL shared-memory file
└── logs/
    └── signal-rs.log
```

The state directory itself is mode `0700`; SQLite files inside default to
`0600` on creation. The path comes from `dirs::data_dir()` - platform-native,
not a hardcoded `~/.local/share/`.

The state directory's permissions must be `0700` and the sled trees within it
default to file mode `0600`. Compromise of this directory equals compromise of
the linked device's identity.

### API Design

**Library - `signal_rs::Client`**

The three operations are: `Client::open` (which is link-or-load), `Client::receive`,
`Client::send`. The fourth public entry, `signal_rs::link`, is a free function
because linking happens before a `Client` exists.

```rust
// First-time linking: creates the state dir if missing, runs provisioning,
// persists identity, uploads initial prekeys, returns the account number.
// Blocks until the primary device scans the QR and the provision message
// arrives (or times out after PROVISIONING_TIMEOUT).
pub async fn link(state_dir: &Path, device_name: &str) -> Result<LinkOutcome, LinkError>;

// Opens an existing state directory. Fails if the dir doesn't exist or the
// identity tree is empty. Does not perform any network I/O until receive() or
// send() is called.
impl Client {
    pub async fn open(state_dir: &Path) -> Result<Self, OpenError>;
}

// Subscribe to incoming envelopes. Returns a Stream. The Stream owns the
// WebSocket connection; dropping the stream closes the connection. Reconnect
// is internal: the stream surfaces a transient error but keeps trying.
impl Client {
    pub async fn receive(&self) -> Result<EnvelopeStream, ReceiveError>;
}

// Send a 1:1 text message. `target` is an E.164 number. To send to self,
// pass the account's own number; Signal fans the message back to all linked
// devices, surfacing it in the phone's Note-to-Self conversation.
impl Client {
    pub async fn send(&self, target: &str, body: &str) -> Result<(), SendError>;
}
```

**CLI - `signal-rs`**

```
signal-rs link --name <DEVICE_NAME>
    Interactive: generates a sgnl:// provisioning URI, prints a QR via the
    qrencode crate (terminal output, no external binary needed), waits for
    the primary device to scan, persists state, exits 0 on success.

signal-rs send <E164> <MESSAGE>
    Smoke-test wrapper around signal_rs::Client::send. Opens the state dir,
    sends the message, exits.

signal-rs receive --once
    Smoke-test wrapper. Opens the state dir, prints the first envelope to
    stdout (as pretty JSON), exits.

signal-rs --help, signal-rs --version
    Standard clap behavior.
```

All three operate against the same state directory: explicit `--state-dir
<PATH>` flag, defaulting to `dirs::data_dir().join("signal-rs")` per
`rules/rust.md`.

### Implementation Plan

Phases are **strictly ordered** - each one depends on the previous. Acceptance
criterion for each phase is the bullet that ends in "`otto ci` passes" or a
specific behavioral test; nothing moves forward with a red CI. Model
annotation per phase: **sonnet** for mechanical scaffolding/wiring, **opus**
for protocol code, novel design, and tricky integrations.

| Phase | Theme | Output |
|---|---|---|
| 1 | Crate reshape + AGPL license | `signal-rs` builds as both lib + bin under AGPL-3.0 |
| 2 | Dependency wiring + libsignal pin verification | `Cargo.toml` references `libsignal-protocol`, `libsignal-net`, `libsignal-net-chat`; minimum commit chosen to include the `libsignal-net-chat` API surface |
| 3 | Storage | `Store` trait + sqlx (SQLite) impl + libsignal storage traits, round-trip tested |
| 4 | ProvisioningCipher port (NEW) | Pure-Rust port of `signal-service-android`'s `ProvisioningCipher` (ECDH + HKDF + AES-CBC + HMAC), unit-tested with known plaintext/ciphertext vectors |
| 5 | Linking | `signal_rs::link` drives `libsignal-net::ProvisioningConnection`, decrypts via Phase 4, persists identity, uploads initial prekeys |
| 6 | Receive | `Client::receive` yields decrypted envelopes via `libsignal-net::chat`, with transactional session-state persistence |
| 7 | Send | `Client::send` wraps `libsignal-net-chat::AuthenticatedChatApi::send_sync_message` / `send_message` |
| 8 | Prekey replenishment | Background task keeps remote prekey count above watermark via `libsignal-net-chat` keys endpoint |
| 9 | CLI smoke-test subcommands | `signal-rs send` and `signal-rs receive --once` |
| 10 | Live-network wiring + manual end-to-end validation | network code lands; manual link → receive → send succeeds against a real Signal account |

#### Phase 2: Dependency wiring + libsignal pin verification
**Model:** sonnet

The original draft of this doc made Phase 2 a go/no-go gate to verify
whether libsignal-rust contained the networking layer at all. The Addendum
at the top of this doc records the result of that verification: yes, it
does - the workspace contains `libsignal-net` (WebSocket transport +
provisioning events) and `libsignal-net-chat` (typed authenticated /
unauthenticated chat APIs including `send_sync_message`). Phase 2 retains
a smaller verification ritual to confirm the *exact pinned commit* contains
those crates with the API surfaces we depend on.

Verification tasks:

- Open `~/repos/signalapp/libsignal` and locate the most recent commit
  whose `rust/net/chat/src/api/messages.rs` defines
  `AuthenticatedChatApi::send_sync_message` and whose
  `rust/net/src/chat/server_requests.rs` defines `ProvisioningEvent`.
  Record the commit SHA. Local checkout is at version `0.94.1`;
  signal-cli pins `0.92.1`, which may predate `libsignal-net-chat` - we
  lead signal-cli's pin if needed.
- Sanity-check that the chosen commit's `Cargo.toml` files build under
  the Rust toolchain pinned in `signalapp/libsignal/rust-toolchain`
  (`1.88` at the local checkout).
- **ProvisionMessage protobuf access path.** Verify whether libsignal
  exports the `ProvisionMessage` Rust struct publicly (via
  `libsignal-protocol::proto::*` or similar) such that Phase 4's
  decryption can decode directly into it, OR whether we must vendor
  Signal's `Provisioning.proto` into our own `build.rs` and generate
  with `prost`. Vendoring is acceptable but creates an upstream-drift
  risk - if Signal adds a new required field, our vendored copy ignores
  it. Pick a path here and document it.
- **Trait injection mechanism for libsignal-net-chat.** Verify that
  `libsignal-net-chat::AuthenticatedChatApi` and its
  Unauthenticated peer accept consumer-provided implementations of
  libsignal's storage traits (`IdentityKeyStore`, `SessionStore`,
  `PreKeyStore`, `SignedPreKeyStore`, `KyberPreKeyStore`) directly,
  OR whether they require libsignal's own concrete in-memory store
  types and force us to write a sync layer between our `SqliteStore`
  and libsignal's in-memory representation. The contract here
  determines how clean Phase 3's storage trait impls can be.
- **Send transport.** Verify how `libsignal-net-chat::AuthenticatedChatApi::send_message`
  (and `send_sync_message`) actually transport their requests:
  (a) over the existing `ChatConnection` WebSocket via multiplexed
      request/response framing,
  (b) over an independent HTTPS request,
  (c) over a separate WebSocket connection.
  Case (c) would be a defect against the "one Client owns one
  WebSocket" model - Signal's servers only tolerate one active
  WebSocket per device, so a second WebSocket from the same Client
  would tear down the receive connection. Case (a) is the assumption
  in the receive/send pipeline diagrams; if reality is (b), the
  diagrams need updating but nothing else changes. Resolve here.

#### Phase 4: ProvisioningCipher port
**Model:** opus

This phase ports `signal-service-android`'s `ProvisioningCipher.java`
into pure Rust. It is the one piece of the Signal stack `signalapp/libsignal`
does NOT provide (verified empirically; see Addendum). libsignal's
`ProvisioningEvent::ReceivedEnvelope` hands us opaque encrypted bytes;
this phase is the Rust code that turns those bytes into a decrypted
`ProvisionMessage` protobuf.

Reference: `signal-service-android` (the canonical Signal client code)
implements `ProvisioningCipher` and its companion `ProvisioningKeyPair`
following Signal's well-specified key-exchange protocol. The Rust port
implements:

1. **Ephemeral keypair generation.** Curve25519 keypair generated by
   `signal_rs::link` and embedded (public half) in the `sgnl://`
   provisioning URI rendered as a QR.
2. **ECDH agreement.** When the encrypted envelope arrives, take the
   peer's public key from the envelope, do X25519 with our ephemeral
   private, and derive a shared secret.
3. **HKDF-SHA256 expansion.** Expand the shared secret with a fixed info
   string (`"TextSecure Provisioning Message"` in the reference code) to
   yield 32 bytes of AES key + 32 bytes of MAC key.
4. **MAC verification.** Compute HMAC-SHA256 over the version byte +
   ciphertext, constant-time compare against the trailing MAC bytes.
   Reject on mismatch.
5. **AES-256-CBC decryption.** Decrypt the ciphertext with PKCS#7 padding
   and the IV embedded at the start of the ciphertext.
6. **Protobuf decode.** Decode the plaintext as a `ProvisionMessage`
   protobuf - the message definition lives in libsignal's protobuf set.

Implementation in `src/crypto/provisioning.rs`:

```rust
pub struct ProvisioningKeyPair { /* Curve25519 */ }

impl ProvisioningKeyPair {
    pub fn generate() -> Self;
    pub fn public_key_bytes(&self) -> [u8; 32];
}

pub fn decrypt_envelope(
    keypair: &ProvisioningKeyPair,
    encrypted: &[u8],
) -> Result<ProvisionMessage, ProvisioningCipherError>;
```

**Critical invariant: Encrypt-then-MAC ordering.** The decrypt flow MUST
verify the HMAC over (version byte + ciphertext) BEFORE attempting to
strip PKCS#7 padding or interpret plaintext. Stripping padding first
introduces a padding oracle - even though this is provisioning-only
(not a continuous wire), the discipline is non-negotiable.

**Memory hygiene.** All ephemeral key material (the
`ProvisioningKeyPair` private half, the ECDH shared secret, the
derived AES key, the derived MAC key) must be wrapped in
`zeroize::Zeroizing<_>` or implement `Drop` via `zeroize` to clear
on scope exit. Rust does not zero memory on drop by default; for
crypto material we add that contract explicitly.

Tests:

- **Known-vector test.** Hard-code a known (ephemeral keypair, peer
  pubkey, ciphertext, expected plaintext) tuple captured from a real
  linking session against a Signal staging account (Phase 10). Assert
  the port produces the expected plaintext bit-for-bit.
- **MAC tamper rejection (ciphertext bit-flip).** Flip a single bit in
  the ciphertext, assert decrypt errors with a MAC failure - not a
  padding failure, not garbage plaintext.
- **MAC tamper rejection (IV bit-flip).** Flip a single bit in the
  IV (which is embedded at the head of the ciphertext blob), assert
  decrypt errors with a MAC failure. Verifies MAC coverage includes
  the IV.
- **Length-boundary safety.** Feed ciphertext of length 0, 15, and 17
  bytes. Assert decrypt returns a clean `ProvisioningCipherError`
  variant in all three cases - never a Rust slice-boundary panic.
  AES-CBC requires multiples of 16 bytes; the port must reject
  malformed input cleanly before it reaches the AES call.
- **Wrong-keypair rejection.** Use a different ephemeral keypair than
  the one that received the envelope; assert decrypt errors with a
  MAC failure (the wrong keypair derives a wrong shared secret,
  which derives wrong AES + MAC keys, which fails the MAC check
  before any plaintext is touched).
- **Padding-without-MAC negative test.** Construct an envelope where
  the MAC is valid but the PKCS#7 padding is malformed (e.g., final
  byte is 0xFF). Assert this fails with a padding-error variant (after
  MAC passes), not a panic and not silent garbage. Confirms the
  encrypt-then-MAC ordering is correct: MAC succeeded, then padding
  rejected.

This phase has no networking and no I/O - it's a focused crypto port
with deterministic test vectors. It is the v0.1 surface's single piece
of novel code; everything else composes existing libsignal API.

#### Phase 5: Linking flow + initial prekey upload
**Model:** opus
- Implement `signal_rs::link` in `src/link.rs`:
  - Call `crypto::provisioning::ProvisioningKeyPair::generate()` (Phase 4).
  - Connect to Signal's provisioning WebSocket via
    `libsignal-net::ProvisioningConnection::connect`.
  - Wait for `ProvisioningEvent::ReceivedAddress { address }`. Build the
    `sgnl://` URI from the ephemeral pubkey + the received address.
  - Invoke the caller's `display_qr` callback with the URI string.
  - Wait for `ProvisioningEvent::ReceivedEnvelope { envelope }`.
  - Call `crypto::provisioning::decrypt_envelope(&keypair, &envelope)`
    (Phase 4) to recover the `ProvisionMessage`.
  - Persist identity, account number (ACI + PNI), device ID, profile key
    into storage via the `Store` trait from Phase 3, and set a
    `link_status` row to `IdentityPersisted`.
  - Generate the initial prekey set (one-time, signed, kyber) and upload
    via Signal's `/v1/devices/{provisioningCode}` (device-completion)
    and `/v2/keys/?identity=aci` (key upload) endpoints. **Note:**
    libsignal-net-chat at the chosen pin does *not* expose these PUT
    endpoints (only `get_pre_keys` on the GET side); signal-rs ships
    them via raw HTTPS through `reqwest` in `src/api.rs`. The wire
    shape is pinned by a snapshot test
    (`api::tests::prekey_upload_body_serializes_to_expected_camel_case_shape`)
    so future drift fails CI.
    On successful upload, transition `link_status` to `Linked`.
    **Linking is not "done" until `link_status == Linked`.**
- **Half-linked state recovery.** If `link` persists identity but crashes
  or returns an error before prekey upload succeeds, the state directory
  is left at `link_status == IdentityPersisted`. `Client::open` MUST
  check `link_status` and refuse to proceed normally if the value is not
  `Linked`. Two recovery paths:
  - Caller invokes `signal_rs::link` again from the same state dir,
    which detects `IdentityPersisted` and resumes by re-generating the
    initial prekey set and uploading it (skipping the protocol exchange
    with the primary - the identity is already established).
  - Or `Client::open` exposes a `resume_link()` method that does the
    same. v0.1 picks one path; default is "link again" because it's
    operator-driven and matches signal-cli's mental model.
  Without this, a partial link leaves the device silently unreachable
  by new peers - Signal cannot establish sessions to a device with no
  prekeys in its keyserver.
- CLI surface: `signal-rs link --name <DEVICE_NAME>` passes a `display_qr`
  callback that renders the QR with `qrcode` to stdout. If the state dir
  is at `IdentityPersisted`, the CLI prints a clear message and resumes
  the prekey upload step instead of re-prompting for a QR scan.
- Integration test: feed a fixture `ProvisionMessage` through the
  Phase-4-decrypted code path, assert correct storage writes. Real
  end-to-end linking against Signal's live servers stays a manual smoke
  test (Phase 10), since it requires a phone scan and cannot be automated.
- Integration test for the half-linked path: simulate a panic between
  identity persistence and prekey upload, then run `link` again and
  confirm it resumes cleanly and leaves the state at `Linked`.

#### Phase 6: Receive pipeline (compose libsignal-net::chat)
**Model:** opus
- `Client::open` constructs the libsignal-net chat connection using the
  stored identity from Phase 3's storage as credentials. We do NOT write
  WebSocket framing, reconnection logic, or chat-service protobuf parsing
  - all that lives in `libsignal-net::chat`. signal-rs consumes the typed
  `ListenerEvent` stream and dispatches on the event kind.
- For each chat envelope received: decrypt via libsignal-protocol's
  session cipher (operating against session records from our `Store`
  trait).
- Implement envelope mapping in `src/envelope.rs`: libsignal's wire
  envelope types -> our public `Envelope` enum (specifically the
  `DataMessage` and `SyncMessage::Sent` variants for v0.1).
- **Receive pipeline must be transactional** (load-bearing for ratchet
  integrity): each envelope is processed as `decrypt -> persist session
  state in a single sqlx::Transaction -> only after the transaction
  commits, send to the broadcast channel`. If we yield before persisting,
  a process crash desyncs the ratchet and breaks every subsequent envelope
  from that peer. Test this explicitly: simulate a panic between decrypt
  and persist, then on next boot confirm the envelope is re-processed
  cleanly.
- **Transaction plumbing through libsignal's storage traits.**
  libsignal-protocol's session-decrypt path takes `&mut dyn
  IdentityKeyStore` / `SessionStore` / `PreKeyStore` and calls
  `load_session` / `store_session` / `take_pre_key` etc. internally.
  These trait methods do not know about `sqlx::Transaction`. If our
  `SqliteStore` holds a bare `SqlitePool`, every internal storage-trait
  call checks out a *new* connection from the pool, which executes
  outside the outer `sqlx::Transaction` we wrapped the envelope
  processing in. **Atomicity is broken.** The fix is a transaction-
  backed store wrapper:

  ```rust
  struct TxStore<'a, 'tx> {
      tx: &'a mut sqlx::Transaction<'tx, sqlx::Sqlite>,
  }

  impl<'a, 'tx> SessionStore for TxStore<'a, 'tx> {
      async fn load_session(&self, addr: &ProtocolAddress)
          -> Result<Option<SessionRecord>, _> {
          sqlx::query("SELECT record FROM sessions WHERE address = ?")
              .bind(addr.to_string())
              .fetch_optional(&mut **self.tx)
              .await
              .map(/* ... */)
      }
      // ... store_session etc. ...
  }
  // (analogous for IdentityKeyStore, PreKeyStore, etc.)
  ```

  The receive loop opens a `sqlx::Transaction`, instantiates a `TxStore`
  borrowing it, calls libsignal's decrypt, then commits. There is real
  lifetime/borrow-checker friction here - `TxStore` mutably borrows
  `tx`, so we cannot interleave other DB work in the same scope.

  An additional borrow-checker hurdle:
  `libsignal_protocol::message_decrypt` takes four disjoint `&mut dyn`
  storage references plus one `&dyn`; pointing all four `&mut`s at the
  same `TxStore` is rejected by the compiler. The implementation
  resolves this by having `TxStore` expose five per-trait sub-stores
  (`TxSessionStore`, `TxIdentityStore`, `TxPreKeyStore`,
  `TxSignedPreKeyStore`, `TxKyberPreKeyStore`), each holding a clone
  of the same `Rc<RefCell<&mut sqlx::Transaction>>`. The compile-time
  borrow checker sees five independent bindings; the runtime
  `RefCell` enforces sequential access (safe because libsignal calls
  the storage traits sequentially with no reentrancy). Atomicity is
  upheld: all five sub-stores commit or roll back together with the
  one underlying transaction.
- **Pool-backed `SqliteStore` is still useful** for operations that
  are NOT inside an envelope-decrypt critical section (initial
  identity persistence during linking, prekey replenishment uploads,
  read-only inspection from the CLI). Two store flavors:
  `SqliteStore` (pool-backed, default) and `TxStore<'_>` (transaction-
  scoped, used only inside the receive loop's decrypt critical
  section).
- On decrypt failure (Bad MAC, unknown prekey, identity-key mismatch):
  log WARN with the peer's address, drop the envelope, do NOT tear down
  the chat connection. Increment a decrypt-failure counter (the metric
  surface itself is out of scope for v0.1 but the increment point must
  exist).
- `Client::receive` returns a Stream backed by a
  `tokio::sync::broadcast` channel; the receive loop is the single
  producer, `receive()` callers are subscribers.
- Integration test using a recorded fixture (libsignal-net wire bytes +
  known ratchet state, decryption asserted to produce the expected
  plaintext). libsignal already has its own tests for the WebSocket
  layer; we only test our composition + the transactional persistence.

#### Phase 7: Send pipeline (compose libsignal-net-chat)
**Model:** opus
- Implement `Client::send`:
  - If no session exists for the target, fetch their prekey bundle via
    `libsignal-net-chat::api::UnauthenticatedChatApi::get_pre_keys` (or
    the authenticated variant if that's the right one - confirm during
    Phase 2 verification) and initialize a new session using
    libsignal-protocol.
  - Encrypt the message via libsignal-protocol's session cipher. Persist
    the resulting session-state update in the same `sqlx::Transaction`
    that records the outbound message timestamp, matching the receive
    pipeline's atomicity rule.
  - Dispatch via `libsignal-net-chat::api::AuthenticatedChatApi::send_message`
    for 1:1 sends, or `send_sync_message` for the Note-to-Self path
    (when `target == own_account_number`).
  - Surface success/failure to caller as `Result<(), SendError>`. Map
    libsignal-net-chat's error enum into our `SendError`, preserving
    actionable details (rate-limit, mismatched-devices, unauthorized).

#### Phase 8: Prekey replenishment
**Model:** opus
- Background task on `Client` triggered by the receive loop: after each
  successful receive batch, poll `libsignal-net-chat`'s keys endpoint
  for our remaining one-time prekey count. If below
  `PREKEY_LOW_WATERMARK`, generate the next batch and upload via the
  same API. We do NOT rely on any server-push notification - the receive
  loop's batch-acknowledgment cycle is the canonical trigger.
- On upload failure (network error, server error): retry with exponential
  backoff. If the remote count reaches zero before we can replenish, log
  ERROR. Inbound messages from peers without an existing session with us
  may be dropped by Signal until prekeys are restored - this is expected
  behavior and the consumer (borg) must tolerate the gap.
- Note: sealed sender is deferred to a later version. For the
  Note-to-Self path (sending to your own number), sealed sender is not
  load-bearing - the receiving device already knows the message is
  from you because it is signed with your account's identity key.

**Scope note (CDSI gap):** the Phase 7 bullet above presumes the
caller already knows the recipient as a `ServiceId` (ACI/PNI). The
v0.1 CLI accepts an E.164 number; resolving E.164 → ACI requires
Signal's attested CDSI service (`libsignal-net::cdsi`, SGX-attested),
which is out of v0.1 scope. The implementation therefore exposes two
send entry points:

- `Client::send_to_aci(aci, body, access_key)` — full 1:1 path. Caller
  must already know the recipient's ACI and `access_key` (16 bytes
  derived from their profile key).
- `Client::send(target, body)` — E.164/Note-to-Self convenience.
  Routes to `send_sync_message` when `target == own_number`; returns
  `SendError::TargetUnsupported` for any other E.164 because the
  CDSI-backed resolver is not in v0.1.

**Model:** sonnet
- `signal-rs send <E164> <MESSAGE>` - calls `Client::send` and exits.
- `signal-rs receive --once` - calls `Client::receive`, prints the first

- Wire `--state-dir`, `--log-level`, `--version`, `--help` plumbing per

**Model:** sonnet
- Run the manual link → receive → send loop against a real Signal account.
- Capture a real provisioning envelope (encrypted bytes + the ephemeral
  keypair that decrypts it) for use as the Phase 4 known-vector test

- Document the procedure in `docs/manual-smoke-test.md` so future hosts can

## Alternatives Considered

### Alternative 1: Use `whisperfish/presage` directly from borg

- **Description:** Skip `signal-rs` entirely. borg adds `presage` as a
  dependency. presage already provides `Manager::link_secondary_device`,
  `Manager::receive_messages`, `Manager::send_message`. borg's hot path
  (1:1 + sync messages) is exactly the subset presage exercises most heavily.
- **Pros:** Smallest possible delta to unblock borg. No new repo to maintain.
  No WebSocket loop, no storage backend, no prekey replenishment, no
  provisioning protocol to implement - all already there. Active upstream.
- **Cons:** Third-party maintained; API has had breaks between minor versions.
  No path toward replacing signal-cli's wider surface. Doesn't address Keegan's
  "the ecosystem is missing a real Rust signal-cli replacement" framing. borg
  inherits presage's roadmap, not its own.
- **Why not chosen:** Scott's explicit fork-choice in brainstorming was
  "Direct-on-libsignal-rust client" - the long-term destination is replacing
  signal-cli, not just unblocking borg. presage stays useful as a reference
  implementation to study, especially for storage schema and provisioning flow.

### Alternative 2: Wrap `signal-cli` daemon (Shape A from reference doc)

- **Description:** Don't write a Rust client. Ship `signal-rs` as a thin Rust
  wrapper around signal-cli's JSON-RPC Unix socket. borg becomes a
  socket-consumer, signal-cli does the actual Signal work.
- **Pros:** signal-cli is mature, broad feature coverage, daemon mode is clean.
  v0.1 ships almost immediately - only JSON-RPC client code.
- **Cons:** JVM tax. Two systemd units. Doesn't move toward replacing
  signal-cli; ENTRENCHES it. The reference doc explicitly calls this the
  fallback shape.
- **Why not chosen:** Same fork-choice. Wrapping the JVM tool is the *opposite*
  of replacing it.

### Alternative 3: Single artifact, library-only (no CLI)

- **Description:** Drop the `[[bin]]`. signal-rs ships only as a library.
  Linking is done by writing a small Rust program that calls `signal_rs::link`.
- **Pros:** Smallest surface.
- **Cons:** The interactive QR ritual is operator-facing, not borg-facing.
  Asking the operator to write a Rust program to link a device is a regression
  vs `signal-cli link`. Every signal-cli user expects a CLI.
- **Why not chosen:** The CLI subcommand for `link` is the one piece a daemon
  consumer (borg) structurally cannot do for itself. It belongs in the CLI.

### Performance

Steady-state baseline targets (from the reference doc § Comparison table):

| Metric | signal-cli (today) | signal-rs (target) |
|---|---|---|
| Resident RSS while connected | 200-300 MB (JVM) | <50 MB |
| CLI process start to `--help` exit | a few seconds (JVM warmup) | <100 ms |
| First envelope after WebSocket connect | tied to JVM warmup + connect | tied to connect only |
| Local decrypt + dispatch per envelope | <100 ms typical | comparable |

These are sanity targets, not SLAs. The point is to confirm the project is
structurally cheaper than the JVM daemon, not to commit to specific numbers.
If `signal-rs` ends up heavier than signal-cli on any of these axes, that's
a defect.

"Cold start" excludes network: a `signal-rs --help` or `signal-rs --version`
invocation does no I/O and should be near-instant; only `link`, `send`, and
`receive --once` pay the network cost.

### Licensing

This is load-bearing. **`signalapp/libsignal` is AGPL-3.0.** Linking against
AGPL code makes the resulting binary a derivative work subject to AGPL
terms - including the "network use is distribution" clause that requires
source disclosure to users interacting with the software over a network.

Consequences:

- `signal-rs` cannot ship under the scaffold's default MIT license. It must
  be AGPL-3.0 (or a compatible license; AGPL is the conservative match).
  Update `LICENSE` and the `Cargo.toml` `license` field in Phase 1.
- `scottidler/second-brain` (which links signal-rs and therefore transitively
  libsignal) inherits the same AGPL flow-through for the borg binary. This
  may or may not be acceptable depending on how second-brain is operated;
  it's a single-operator personal tool today, so source-disclosure to
  "network users" maps to "Scott discloses to himself" - operationally a
  non-issue, but the license metadata must be correct.
- `tatari-tv/*` consumers (if any future Tatari project wants to use
  signal-rs) need to be aware of the AGPL implications before adopting.
  This is a separate conversation with legal that does not block v0.1.

For comparison, `signal-cli` itself is GPL-3.0 - also copyleft, but without
the network clause. AGPL is the strict superset; choosing AGPL keeps
signal-rs flow-through-compatible with libsignal without further analysis.

### Security

The state directory holds the device's identity keypair and Double Ratchet
state for every peer the user has talked to. Compromise of the directory
equals compromise of the linked device.

Mitigations:

- State dir created with `0700` permissions on initial link.
- sled trees default to `0600`.
- No state ever written to stdout / stderr / log (the log file lives outside
  the state dir).
- Provisioning keypair is generated fresh per link, used once, then discarded
  (it does not become the device's long-term identity).
- Linked-device deauth: if the user removes the device from their phone's
  Linked Devices list, Signal stops fanning envelopes to it. signal-rs detects
  this via WebSocket auth failure and surfaces it as a clear error rather than
  silently failing.
- We do **not** implement identity-key trust pinning in v0.1. New peers'
  identity keys are accepted on first sight. For borg's single-operator
  Note-to-Self use case, the only peer that matters is the user's own account,
  and that identity is established during link. Trust pinning for arbitrary
  contacts is a later-version concern.

### Edge cases the design handles explicitly

These are behaviors the implementation must get right; each maps to a test
or a documented surface.

- **Running `signal-rs link` against an already-linked state directory.**
  Refuse with a clear error rather than silently overwriting; overwriting
  would orphan the previous device's identity and the previous device's
  remaining prekeys in Signal's keyserver. CLI exit code is non-zero.
- **Running `Client::open` against an empty / unlinked state directory.**
  `OpenError::NotLinked` - caller must run `link` first. Distinguishes
  cleanly from `OpenError::Corrupt` or `OpenError::Io`.
- **Two `signal-rs` processes touching the same state dir.** Reference doc
  flags this as the "two daemons desync the ratchet" gotcha. Mitigation:
  hold an exclusive `flock(2)` on `<state_dir>/store.db` at `Client::open`;
  the second opener gets `OpenError::AlreadyOpen` immediately.
- **Device deauthorized from the primary's Linked Devices list.** Signal's
  chat-service WebSocket auth fails on next connect. Surface as
  `ReceiveError::Deauthorized` / `SendError::Deauthorized`. Do not retry -
  the only recovery is `link` again. Document that this is a terminal state
  for the current state dir.
- **Extended offline period (> Signal's ~24-48h queue retention).** Messages
  sent during the window are lost from Signal's servers and will never
  arrive. signal-rs cannot fix this; the consumer (borg) must know it
  inherits this property. Documented as a non-feature, not a defect.
- **Public-type breaking changes between v0.1 and v0.2.** All public enums
  and structs are `#[non_exhaustive]`; new variants and fields can be added
  without bumping the major version. SemVer policy: minor bumps for additive
  changes, major bumps only for breaking type renames or trait surface
  changes.
- **Storage WAL growth.** SQLite WAL grows on hot writes. WAL checkpoint
  happens automatically on connection close; for long-running daemons
  (borg holds Client for its whole lifetime), run a periodic
  `PRAGMA wal_checkpoint(TRUNCATE)` from a background task.
- **Ratchet atomicity on receive.** The receive pipeline must persist
  updated session state BEFORE yielding the decrypted envelope to the
  consumer. If we yield first and then crash mid-persistence, the next
  process boot replays the envelope with stale ratchet state and gets a
  "Bad MAC" / decrypt failure on every subsequent envelope from that peer.
  The correct ordering is: `decrypt -> persist session record (and any
  prekey consumption) in a single sqlx::Transaction -> yield to
  subscriber`. The broadcast channel's send happens only after the
  transaction commits.
- **Decrypt failure (Bad MAC, unknown prekey, identity-key mismatch).**
  Log the failure with the peer's address at WARN level, drop the offending
  envelope, do NOT tear down the WebSocket - one broken session must not
  kill the whole connection. Surface the failure count via a future metrics
  surface (out of scope for v0.1, but the code path must increment a
  counter that exists for instrumentation). Smarter recovery (session
  reset negotiation with the peer) is deferred to a later version; v0.1
  baseline is "log + drop."
- **Prekey exhaustion.** Signal's keys endpoint reports our remaining
  one-time prekey count after each successful receive batch. When the
  count falls below `PREKEY_LOW_WATERMARK`, the prekey worker generates
  and uploads more. We do NOT rely on any server-push notification - the
  receive loop's batch acknowledgment is the canonical trigger. If
  replenishment fails (network error during upload), retry with exponential
  backoff; if our remote count hits zero, log ERROR and continue - inbound
  messages from peers that don't yet have a session with us will be
  dropped by Signal until prekeys are restored.

### Testing Strategy

- **Unit tests** in each module: storage round-trips, envelope deserialization,
  CLI argument parsing.
- **Crypto correctness tests**: use libsignal's own test vectors where
  applicable to confirm we're using the library correctly.
- **The load-bearing filter test** (per reference doc § "Note to Self is the
  ingest channel"): given a self-sync, a sync-to-friend, and a dataMessage,
  assert exactly one passes the `destination == account_number` filter. This
  test lives in borg in second-brain, but signal-rs's envelope types must
  expose `destination` correctly for that test to be writable.
- **Integration tests** against recorded wire fixtures: a decrypted envelope
  with known plaintext, encrypted with known ratchet state - confirms our
  decryption path matches libsignal.
- **Manual smoke test** for link/send/receive against a real Signal account.
  Procedure documented in `docs/manual-smoke-test.md`. There is no headless
  end-to-end test for linking because linking requires a real phone scan.

### Rollout Plan

- Tag the v0.1.0 release once Phase 1-9 are green and the manual smoke test
  passes against a real account.
- second-brain consumes `signal-rs` via git tag (not crates.io for v0.1 -
  publishing to crates.io can happen once the surface is more stable).
- The Shape A signal-cli daemon path stays operable in the meantime; nothing
  about this design forces second-brain to switch off signal-cli until
  signal-rs's v0.1 is in hand.
